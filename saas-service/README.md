# ChuHai SaaS Service

## Overview

This is the SaaS (Software as a Service) version of the ChuHai demand discovery platform. Unlike the CLI version which is designed for individual developers and researchers, this service provides a multi-tenant web API for businesses to integrate demand discovery capabilities into their workflows.

## Technology Stack

- **Backend**: FastAPI (Python 3.12)
- **Database**: PostgreSQL 16 with PostGIS extension
- **Caching/Broker**: Redis 7
- **Task Queue**: Celery with Redis broker
- **API Documentation**: Swagger UI (automatically generated by FastAPI)
- **Containerization**: Docker & Docker Compose
- **Code Quality**: Black, Ruff, MyPy
- **Testing**: Pytest

## Quick Start (Docker Compose)

1. Clone the repository:
   ```bash
   git clone <repository-url>
   cd saas-service
   ```

2. Copy the example environment file:
   ```bash
   cp .env.example .env
   ```

3. Edit the `.env` file to set your actual API keys and secrets.

4. Start the services:
   ```bash
   docker compose up -d
   ```

5. The API will be available at http://localhost:8000
   - API documentation: http://localhost:8000/docs
   - ReDoc documentation: http://localhost:8000/redoc
   - pgAdmin (if enabled): http://localhost:5050

## Environment Variables

Refer to `.env.example` for a complete list of environment variables. Key variables include:

- `DATABASE_URL`: PostgreSQL connection string
- `REDIS_URL`: Redis connection string
- `SECRET_KEY`: Secret for JWT token generation
- `SENSENOVA_API_KEY`: API key for SenseNova services
- `GLM_API_KEY`: API key for Zhipu AI (GLM) services
- `ANTHROPIC_API_KEY`: API key for Anthropic Claude
- `STACKEX_API_KEY`: Optional API key for StackExchange API
- `CORS_ORIGINS`: List of allowed origins for CORS
- `LOG_LEVEL`: Logging level (debug, info, warning, error)
- `WORKER_CONCURRENCY`: Number of Celery worker processes

## API Documentation

Once the service is running, interactive API documentation is available at:
- Swagger UI: http://localhost:8000/docs
- ReDoc: http://localhost:8000/redoc

## Development Guide

### Prerequisites

- Docker and Docker Compose
- Python 3.12+ (for local development without Docker)
- Git

### Local Development (Without Docker)

1. Create a virtual environment:
   ```bash
   python -m venv venv
   source venv/bin/activate  # On Windows: venv\Scripts\activate
   ```

2. Install dependencies:
   ```bash
   pip install -e ".[dev]"
   ```

3. Set up pre-commit hooks (optional but recommended):
   ```bash
   pre-commit install
   ```

4. Run the application:
   ```bash
   uvicorn app.main:app --reload
   ```

5. Run tests:
   ```bash
   pytest
   ```

### Database Migrations

We use Alembic for database migrations. To create a new migration:

```bash
alembic revision --autogenerate -m "description"
```

To apply migrations:
```bash
alembic upgrade head
```

## 本地开发

### 使用 Docker Compose 启动（推荐）

```bash
# 1. 进入项目目录
cd saas-service

# 2. 复制环境变量模板
cp .env.example .env

# 3. 编辑 .env 文件，设置必要的 API Keys
# 必填：SENSENOVA_API_KEY, GLM_API_KEY, ANTHROPIC_API_KEY

# 4. 启动所有服务（后台运行）
docker compose up -d

# 5. 查看服务状态
docker compose ps

# 6. 查看日志
docker compose logs -f api
```

服务启动后可访问：
- API: http://localhost:8000
- Swagger UI: http://localhost:8000/docs
- ReDoc: http://localhost:8000/redoc
- pgAdmin: http://localhost:5050 (默认账号: admin@chuhai.com / admin)

### 本地开发（无 Docker）

```bash
# 1. 创建虚拟环境
python -m venv venv
source venv/bin/activate  # Windows: venv\Scripts\activate

# 2. 安装依赖
pip install -e ".[dev]"

# 3. 启动 PostgreSQL 和 Redis（需自行安装或使用 Docker）
# 仅启动数据库和缓存：
docker compose up -d postgres redis

# 4. 运行数据库迁移
alembic upgrade head

# 5. 启动 API 服务
uvicorn app.main:app --reload

# 6. 另开终端启动 Celery Worker
celery -A app.workers.celery_app worker -l info
```

## 常见问题

### 1. 数据库连接失败
**错误**: `sqlalchemy.exc.OperationalError: could not connect to server`
**解决**:
- 确保 PostgreSQL 容器正在运行：`docker compose ps postgres`
- 检查 `.env` 中的 `DATABASE_URL` 是否正确
- 确认网络连通性：`docker compose exec api ping postgres`

### 2. Redis 连接失败
**错误**: `redis.exceptions.ConnectionError: Error 111 connecting to localhost:6379`
**解决**:
- 确保 Redis 容器正在运行：`docker compose ps redis`
- 检查 `.env` 中的 `REDIS_URL` 和 `CELERY_BROKER_URL`
- 本地开发时需单独启动：`docker compose up -d redis`

### 3. 模块导入错误
**错误**: `ModuleNotFoundError: No module named 'app'`
**解决**:
- 确保在 `saas-service` 目录下运行命令
- 检查虚拟环境是否激活且依赖已安装：`pip install -e ".[dev]"`
- 运行验证脚本排查：`python scripts/test_startup.py`

### 4. 端口冲突
**错误**: `Address already in use` 或 `port 8000 already allocated`
**解决**:
- 停止占用端口的进程：`lsof -ti:8000 | xargs kill -9` (Linux/Mac)
- 或修改 `docker-compose.yml` 中的端口映射
- Windows: `netstat -ano | findstr :8000` 找到 PID，然后 `taskkill /PID <PID> /F`

### 5. 数据库迁移失败
**错误**: `alembic.util.exc.CommandError: Target database is not up to date`
**解决**:
- 先回滚再重新应用：`alembic downgrade base && alembic upgrade head`
- 或删除数据库重建：`docker compose down -v && docker compose up -d postgres && alembic upgrade head`
- 检查模型定义是否与数据库同步

## Deployment Checklist (Production)

1. **Security**
   - [ ] Change all default passwords and secrets
   - [ ] Use strong, randomly generated `SECRET_KEY`
   - [ ] Set `LOG_LEVEL` to `warning` or `error` in production
   - [ ] Consider enabling HTTPS termination at reverse proxy level
   - [ ] Restrict CORS origins to specific domains
   - [ ] Disable pgAdmin service or restrict access to internal network

2. **Performance**
   - [ ] Adjust `WORKER_CONCURRENCY` based on server CPU cores
   - [ ] Monitor Redis memory usage and set appropriate maxmemory policy
   - [ ] Consider using managed PostgreSQL and Redis services for better reliability
   - [ ] Enable connection pooling in database configuration

3. **Reliability**
   - [ ] Set up automated backups for PostgreSQL data
   - [ ] Configure log rotation
   - [ ] Set up monitoring and alerting (CPU, memory, disk, response times)
   - [ ] Consider using a process manager like systemd or supervisor for Docker containers
   - [ ] Set up health checks in your orchestration platform (Kubernetes, Docker Swarm, etc.)

4. **Scalability**
   - [ ] For high traffic, consider running multiple API replicas behind a load balancer
   - [ ] Separate Celery worker pools for different task types (e.g., high priority vs low priority)
   - [ ] Use Redis clustering for Redis if needed

5. **Maintenance**
   - [ ] Keep dependencies updated regularly
   - [ ] Monitor for security vulnerabilities in dependencies
   - [ ] Regularly audit API keys and rotate them periodically
   - [ ] Document any customizations or extensions to the base service
